# Звіт про виконання практичної роботи: Налаштування Git-workflow для аналізу даних

## 1. Опис проєкту
Проєкт **open-data-ai-analytics** реалізований для демонстрації навичок управління версіями у проєктах з аналізу даних. Основна мета полягала у створенні структурованого репозиторію, роботі з гілками та успішному вирішенні конфліктів.

* **Назва проєкту:** Аналіз відкритих даних [Вкажіть вашу тему]
* **Джерело даних:** [Вкажіть посилання на датасет]
* **Гіпотези для аналізу:**
    1. **Як змінюється кількість поїздок залежно від часу доби та дня тижня?**
    2. **Чи існує залежність між відстанню поїздки та її вартістю?**
    3. **Які райони Нью-Йорка мають найбільшу кількість поїздок?**

---

## 2. Структура репозиторію
Для проєкту було створено стандартну архітектуру, що забезпечує чіткий поділ даних, коду та звітів:
* `data/` — сирі та оброблені дані (ігноруються за великого розміру).
* `notebooks/` — Jupyter-нотатники для експериментів.
* `src/` — основні Python-скрипти для обробки та аналізу.
* `reports/figures/` — візуалізації та результати аналізу.



---

## 3. Налаштування Git та Feature Branch Workflow
У роботі використано модель розгалуження **Feature Branching**. Кожен етап розробки виконувався в окремій гілці:
1. `feature/data_load` — реалізація скриптів завантаження.
2. `feature/data_quality_analysis` — перевірка даних на пропуски та аномалії.
3. `feature/data_research` — побудова моделей та основний аналіз.
4. `feature/visualization` — побудова графіків.

Для запобігання засміченню репозиторію було налаштовано файл `.gitignore`, який ігнорує кеш-файли Python, віртуальні середовища та великі масиви даних у `data/raw/`.

---

## 4. Створення та розв'язання конфлікту злиття
Згідно з завданням, було штучно створено конфлікт у файлі `README.md`.

**Процес:**
1. Одночасно змінено одну і ту саму секцію в гілці `feature/data_research` та у master.
2. При спробі злиття другої гілки в `main` Git видав помилку `CONFLICT (content)`.

**Вирішення:**
Конфлікт був вирішений шляхом ручного редагування маркерів конфлікту:
```text
<<<<<<< HEAD
(Зміни з першої гілки)
=======
(Зміни з другої гілки)
>>>>>>> feature/data_research
```
Було прийнято рішення об’єднати обидві секції тексту, щоб зберегти цілісність документації, після чого зміни були додані через `git add` та зафіксовані фінальним комітом злиття.

## 5. Версіонування та історія
По завершенню всіх етапів розробки було виконано наступні кроки для стабілізації релізу:

* Додано файл **CHANGELOG.md** із детальним описом усіх внесених змін та нових функцій.
* Створено анотований тег версії **v0.1.0** за допомогою команди:
    ```bash
    git tag -a v0.1.0 -m "Перший реліз проєкту: налаштована база та аналітичні модулі"
    ```

## 6. Візуалізація графа Git
Нижче наведено дерево комітів, що демонструє паралельну розробку гілок, момент виникнення конфлікту та фінальну структуру репозиторію:

```bash
git log --oneline --graph --decorate --all
```

Під час виконання роботи було закріплено практичні навички роботи з **Git** у контексті Data Science проєкту. Налаштовано процес паралельної розробки через **Feature Branches**, успішно реалізовано стратегію вирішення **merge-конфліктів** та закладено основу для відтворюваності результатів аналізу через чітку структуру репозиторію та контроль версій.
